# x402 / USDC

The [x402](https://github.com/0xKoda/x402-wallet/blob/main/wallet.md) rail settles a **real
USDC payment on base mainnet** and returns the `SettlementResponse` + `PaymentRequirements`
the gateway extracts the metered cost from ‚Äî metering into the **same cross-rail cap as
Stripe**. This is the second rail and the moat. **Real money is the default**: with no
`--test-mode` flag the rail resolves to **base mainnet** and moves **real USDC** on-chain.

!!! danger "Real money is the default"
    With no `--test-mode`, this rail settles **real USDC on base mainnet**. On-chain USDC
    transfers are **irreversible** ‚Äî there is no chargeback. The `--budget` cap is the hard
    ceiling ‚Äî an agent provably cannot spend past it ‚Äî and it is **mandatory**. Start with the
    smallest cap that proves the point, and prove the cap in [testnet mode](#testnet-mode)
    first.

!!! warning "x402 needs a funded wallet + facilitator ‚Äî not just a key"
    Unlike Stripe, x402's on-chain settle needs **a funded base-mainnet USDC wallet *and* an
    x402 facilitator URL**. A key alone does **not** fund the transfer. This is real setup;
    budget for it.

## Prerequisites

- A **base mainnet** wallet **funded with real USDC** (private key, `0x‚Ä¶`), plus a little ETH
  on base mainnet for gas.
- A **mainnet** x402 facilitator URL (the service that brokers the settle).

## Setup

```bash
# settle live on base mainnet, with the wallet + facilitator custodied by the gateway
export X402_WALLET_PRIVATE_KEY=0x...        # a FUNDED base MAINNET USDC wallet
export X402_FACILITATOR_URL=https://...     # a mainnet x402 facilitator
auths-mcp wrap \
  --scope paid.call --budget '$5' --ttl 30m \
  --custody-credential X402_WALLET_PRIVATE_KEY \
  --custody-credential X402_FACILITATOR_URL \
  -- node examples/payments/adapters/x402-adapter/server.mjs
```

Two things are non-negotiable here:

- **The cap is mandatory.** Omit `--budget` and the gateway refuses to wrap the rail at all ‚Äî
  `budget-required`, fail-closed, before any rail is touched. Real money with a skippable cap
  is not an allowed shape.
- **The mode is disclosed.** Every payment-rail wrap prints a `mode=real` banner and a
  `mode=real ‚Ä¶` machine line at startup, so a live rail is never silent. Run
  `auths-mcp wrap --show-mode ‚Ä¶` to resolve and disclose the mode **without** serving the
  proxy or settling on-chain.

## Custody ‚Äî the agent never holds the wallet

The wallet private key and facilitator URL live in the gateway's custody vault
(`--custody-credential`), never with the agent. The agent holds only its scoped,
budget-bound delegation. An agent that tries to settle without going through the gateway has
**no wallet** ‚Äî so the boundary is unbypassable.

## How it works

```
agent ‚îÄ‚îÄtools/call (x402)‚îÄ‚îÄ‚ñ∂ gateway ‚îÄ‚îÄtools/call‚îÄ‚îÄ‚ñ∂ x402-adapter ‚îÄ‚îÄ‚ñ∂ facilitator (base mainnet)
                                ‚îÇ  reserve ceiling BEFORE the rail, ACROSS rails  ‚óÄ‚îÄ‚îÄ SettlementResponse
                                ‚îÇ  extract requirements.maxAmountRequired (atomic USDC ‚Üí cents)
                                ‚îÇ  settle into the SAME cap the Stripe rail meters into
                                ‚ñº
                             receipt: rail=x402, mode=real, tx=0x‚Ä¶, cross-rail total
```

### atomic-USDC ‚Üí cents

USDC has **6 decimals**, so x402 amounts are **atomic**: `1500000` atomic = `1.50` USDC =
**150 cents**. The conversion is `atomic / 10000`, **exact only** ‚Äî a sub-cent residue is
refused, not truncated, so the metered cost equals the settled amount exactly. The receipt
names the on-chain settlement **tx** (`0x‚Ä¶`).

### The cross-rail moat

A `$1.50` Stripe charge **+** a `$1.50` x402 settle = `$3.00` of one `$5` cap. A later
**`$0.60`** x402 settle is tiny in a per-rail x402 silo (x402-alone is only `$2.10`) ‚Äî but
summed cross-rail onto `$4.50` it would reserve to `$5.10 > $5`, so the gateway refuses it
`usage-cap-exceeded` **before** the adapter settles. Two siloed per-rail budgets would each
wave it through; the one cross-rail counter refuses it.

## Never faked

With no wallet + facilitator the adapter runs against the recorded settlement shape (not a
fabricated on-chain settle), so it stays runnable and self-checkable
(`node examples/payments/adapters/x402-adapter/test.mjs`). If the live path is reached
without a reachable facilitator it **throws** rather than inventing a transaction.

## Troubleshooting

| Symptom | Cause |
| --- | --- |
| gateway refuses to wrap (`budget-required`) | no `--budget` ‚Äî the cap is mandatory for a payment rail (both modes) |
| adapter runs but never settles on-chain | no wallet + facilitator ‚Üí it returns the recorded settlement shape |
| live path throws instead of settling | facilitator URL unset/unreachable ‚Äî by design, it never fabricates a tx |
| `usage-cap-exceeded` on a tiny x402 call | the cross-rail moat ‚Äî the shared cap is already spent (expected) |
| sub-cent amount refused | x402 atomic‚Üícents is exact-only; align the amount to whole cents |
| unsure which network is live | run `auths-mcp wrap --show-mode ‚Ä¶` ‚Äî it prints `mode=real|test` (base vs base-sepolia) and exits |

## Testnet mode

To exercise the rail **without real money**, opt into sandbox rails with `--test-mode` (or
`AUTHS_MCP_TEST_MODE=1` for the adapter). Testnet mode resolves the x402 rail to
**base-sepolia** with **testnet USDC** ‚Äî no real money. The cap is **still mandatory**, and
the mode is disclosed as `mode=test`.

```bash
export X402_WALLET_PRIVATE_KEY=0x...        # a FUNDED base-sepolia USDC testnet wallet
export X402_FACILITATOR_URL=https://...     # a base-sepolia x402 facilitator
auths-mcp wrap \
  --test-mode \
  --scope paid.call --budget '$5' --ttl 30m \
  --custody-credential X402_WALLET_PRIVATE_KEY \
  --custody-credential X402_FACILITATOR_URL \
  -- node examples/payments/adapters/x402-adapter/server.mjs
```

Fund the base-sepolia wallet from a base-sepolia ETH faucet (for gas) and a testnet USDC
faucet. Use testnet mode to prove the cap refuses the over-cap call before you ever point a
funded mainnet wallet at the rail.
